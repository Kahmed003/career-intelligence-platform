## Purpose

This document defines the canonical object types represented by Career OS.

It specifies:

* Why each object exists.
* Whether it is first-class or supporting.
* Which architectural layer owns it.
* Its core properties.
* Its expected capabilities.
* Its major relationships.
* Its AI behavior.
* Its validation and identity rules.
* Its lifecycle reference.

This document intentionally does not define final database tables, indexes, API payloads, or user-interface layouts. Those will be derived later.

---

# 1. Object Classification

Career OS uses three architectural layers.

## Personal Layer

Objects unique to the user and their professional journey.

Examples:

* Goal
* Decision
* Project
* Application
* Knowledge Item
* Document
* Asset
* Time Block

## World Layer

Objects that exist independently of the user.

Examples:

* Person
* Organization
* Opportunity
* Skill
* Evidence Source
* Visa Pathway
* Country

## Intelligence Layer

Objects generated by Career OS through analysis.

Examples:

* Insight
* Recommendation
* Score
* Career Risk
* Daily Mission
* Weekly Strategy

Objects are also classified as either:

* **First-class objects:** receive dedicated identity, search visibility, history, relationships, and AI reasoning.
* **Supporting records:** support execution or history but do not normally receive their own top-level navigation experience.

# 2. Shared First-Class Object Contract

Unless explicitly excluded, every first-class object should support the following conceptual contract.

## Identity fields

* `id`
* `object_type`
* `title` or canonical display name
* `slug` where human-readable URLs are useful
* `created_at`
* `updated_at`
* `archived_at`
* `created_by`
* `source_type`

## Ownership fields

* `layer`

  * personal
  * world
  * intelligence
* `owner_user_id` where applicable
* `visibility`

  * private
  * system
  * future_shared

## Shared capabilities

* Timeline
* Related objects
* Tags
* Search
* Attachments
* Activity history
* AI insights
* Archival
* Provenance
* Notes or linked Knowledge Items

## Shared validation rules

1. Every first-class object must have a stable unique identifier.
2. Every object must belong to exactly one architectural layer.
3. Personal objects must identify their owner.
4. World objects should preserve provenance when created from external sources.
5. Intelligence objects must identify the evidence and reasoning that produced them.
6. Archived objects remain retrievable unless permanently deleted by the user.
7. AI-generated values must be distinguishable from user-entered values.
8. System timestamps must not be used as substitutes for real-world event dates.

# 3. First-Class Personal Objects

## 3.1 Goal

### Definition

A desired future outcome that the user intends to achieve.

### Purpose

Goals give Career OS a strategic frame for prioritization, recommendations, progress tracking, and decision support.

### Examples

* Secure a strong internship by summer 2027.
* Obtain a full-time role by summer 2028.
* Gain admission to a fully funded PhD program.
* Become a stronger programmer.
* Build a high-quality technical portfolio.

### Core properties

* `id`
* `title`
* `description`
* `goal_type`
* `status`
* `priority`
* `target_date`
* `start_date`
* `success_criteria`
* `motivation`
* `progress_method`
* `progress_value`
* `confidence_level`
* `owner_user_id`
* `created_at`
* `updated_at`

### Suggested goal types

* career
* graduate_school
* research
* skill
* portfolio
* networking
* finance
* immigration
* personal_development

### Capabilities

* Milestones
* Progress history
* Supporting Projects
* Supporting Opportunities
* Required Skills
* Required Assets
* Risks
* AI recommendations
* Timeline
* Linked Decisions

### AI behavior

Career OS may:

* Detect stalled progress.
* Identify missing prerequisites.
* Suggest milestones.
* Compare actual progress against target dates.
* Recommend Projects and Opportunities.
* Surface conflicts between Goals.
* Explain why an action supports or does not support the Goal.

### Validation rules

* A Goal must have a title.
* A Goal should have measurable success criteria.
* A strategic Goal should have a target date or explicit open-ended status.
* Progress may not exceed its defined maximum.
* AI-generated progress estimates must be labeled as inferred.

Defined in `LIFECYCLES.md`.

* Identify shared interests.
* Recommend conversation topics.

### Validation rules

* Health scores must remain separate from user-entered judgments.
* Human trust should not be reduced to a single definitive number.

# 7. Object Identity and Deduplication

## Person identity

Potential matching signals:

* Name
* Email
* Organization
* Professional profile
* Publication history

Career OS should suggest merges but require confirmation where ambiguity exists.

## Organization identity

Potential matching signals:

* Canonical name
* Website domain
* Headquarters
* Parent Organization
* External identifier

## Opportunity identity

Potential matching signals:

* Organization
* Title
* Location
* Deadline
* Source URL
* External posting identifier

## Document identity

A new revision should normally become a Document Version rather than a duplicate Document.

## Knowledge identity

Semantically similar Knowledge Items may be linked or suggested for consolidation, but should not be merged automatically when meaning differs.

# 8. Archival and Deletion

## Archive

Archiving removes an object from normal active views while preserving history and relationships.

## Soft delete

Soft deletion hides an object while allowing recovery.

## Permanent deletion

Permanent deletion removes the object and must account for:

* Attachments
* Intelligence outputs
* Relationship edges
* Audit requirements
* External integration identifiers

Personal data deletion must take precedence over analytics retention.

# 9. AI Authority Boundaries

Career OS may:

* Classify information.
* Extract structured fields.
* Suggest relationships.
* Generate Insights.
* Generate Recommendations.
* Draft content.
* Propose Tasks.
* Suggest calendar blocks.
* Flag Risks.

Career OS may not, without explicit approval:

* Send emails.
* Submit applications.
* Accept offers.
* Execute trades.
* Alter external Calendar Events.
* Change immigration records.
* Make final Decisions on behalf of the user.
* Permanently delete user data.
